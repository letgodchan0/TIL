<br>

>  절차지향과 객체지향은 컴퓨터 프로그래밍 패러다임의 대표적인 방식

절차지향 : 데이터와 처리 방법을 분리하여 함수들의 모음으로 이루어진 순차적인 절차로 구성하는 방식 

- 프로그램을 함수들의 모음으로 이루어진 순차적인 절차로 구성하며, 데이터에 대한 처리 방식을 중심으로 설계한다. (C언어)

- 장점 : 컴퓨터 처리 구조와 유사해서 실행 속도가 빠름.

- 단점 : 코드 재사용성이 낮고, 유지보수가 어렵다.

 <br>

객체지향 : 데이터와 처리 방법을 하나의 단로 묶어 객체들의 간의 상호작용으로 구성하는 방식


- 데이터와 처리 방법을 하나의 단위로 묶는 방식 (자바, 파이썬, C++), 프로그램을 객체들의 모임으로 이루어진 상호작용으로 구성하며, 객체들은 상태(데이터)와 행위(메서드)를 가지고 있다.
- 장점 : 코드 재사용성이 높고, 유지보수가 용이하다.
- 단점 : 처리 속도가 느리다.

### 객체지향의 특징에 대해 설명하세요.

<details>
   <summary> 준비 답안 보기 (👈 Click)</summary>
<br />

> 데이터와 기능을 하나의 단위로 묶어서 객체를 만들고, 객체의 상호작용을 통해 문제를 해결하는 방식

<br>

추상화 - 객체의 공통된 속성이나 기능을 추출하여 정의하는 것

- 예를 들어, 자바에서는 객체를 추상화해서 클래스를 만든다고 표현한다. 실제 객체가 존재하고, 그러한 객체들의 공통된 속성과 기능만 뽑아서 가지고 있는 클래스가 있다. 또 구조체의 선언처럼 어떠한 정보를 묶음으로 표현할 때, 우리가 필요한 정보들의 형태만 모아서 선언해주는 것도 추상화이다.

<br>

 캡슐화 - 내부의 동작 방법을 숨기고, 사용자에게는 사용 방법만을 노출시키는 것

- 중요 정보를 숨겨서 잘못된 수정을 막고(정보은닉), 사용자가 쉽게 사용할 수 있도록 만든다.
- 예를 들어, 함수를 사용할 때 사용하는 외부에서는 그 함수에 어떤 값을 주면 어떤 값을 결과로 받아오는 것만 알지 그 함수 내부가 어떤 코드로 동작하는 지는 모른다. private 변수에 접근할 때 getter나 setter를 통해 간접적으로 접근하는 것도 캡슐화와 연관된다.
- 한 곳에 변화가 일어나도 다른 곳에 미치는 영향을 최소화 시키는 것
- 외부로부터 클래스에 정의된 속성과 기능들을 보호하고, 필요한 부분만 외부로 노출될 수 있도록 하여 각 객체 고유의 독립성과 책임 영역을 안전하게 지키고자 하는 목적이 있습니다.

<br>

 상속 -  코드의 재사용성을 극대화하기 위한 장치로, 부모 클래스의 속성을 물려받는 것, 설정에 따라 완전히 같지 않을 수도 있다.

- 기존의 클래스를 재활용하여 새로운 클래스를 작성하는 자바의 문법 요소

<br>

다형성 - 하나의 클래스나 메소드가 다양한 방식으로 동작이 가능한 것을 의미하며, 오버라이딩과 오버로딩이 있다. 

- 한 타입의 참조 변수를 통해 여러 타입의 객체를 참조할 수 있도록 하는 것!!!!
  - 상위 클래스 타입의 참조 변수로 하위 클래스의 객체를 참조할 수 있도록 하는 것!!

- 하나의 타입만으로 여러 가지 타입의 객체를 참조할 수 있는 것
- 어떤 객체의 속성이나 기능이 상황에 따라 여러 가지 형태를 가질 수 있는 것
- 인스턴스를 만들 때 하나의 클래스가 다양한 얼굴을 갖게 하는 것
- 부모 클래스의 메소드를 자식 클래스가 오버라이딩해서 자신의 역할에 맞게 활용하는 것



</details>

### 다형성과 인터페이스를 연관지어 설명하세요.

<details>
   <summary> 준비 답안 보기 (👈 Click)</summary>
<br />

다형성 : 하나의 메소드나 클래스가 다양한 방법으로 동작하는 것

인터페이스 : 클래스 혹은 프로그램이 제공하는 기능을 명시적으로 선언하는 역할

- oop에서 인터페이스는 어떤 객체의 역할만을 정의하여 객체들 간의 관계를 유연하게 연결하는 역할
- 인스터스를 생성할 수 없다.
- 메서드의 선언만 있고, 구현체가 없는 추상 메서드와 상수를 가지고 있다. 

인터페이스를 구현하는 클래스는 인터페이스의 메서드를 각자의 방식으로 구현할 수 있기 때문에, 같은 인터페이스를 구현하는 다양한 클래스가 서로 다른 방식으로 동작할 수 있다. 

예를 들어, 동물(Animal) 인터페이스가 있다고 가정해 봅시다. 이 인터페이스는 울음소리를 내는 cry() 메서드를 선언합니다. 각 동물은 다른 울음소리를 내기 때문에, 이 인터페이스를 구현한 클래스는 각자의 방식으로 cry() 메서드를 구현할 수 있습니다. 예를 들어, 개 클래스는 "멍멍" 소리를 내는 cry() 메서드를 구현하고, 고양이 클래스는 "야옹" 소리를 내는 cry() 메서드를 구현할 수 있습니다. 따라서 동물 인터페이스를 사용하는 코드에서는 각 클래스가 다른 방식으로 cry() 메서드를 호출할 수 있어서, 다형성이 구현됩니다.

</details>

### 추상 클래스와 인터페이스의 차이점

<details>
   <summary> 준비 답안 보기 (👈 Click)</summary>
<br />

- 추상 클래스는 상속을 받아 기능을 이용하거나 확장시키는 목적이고, 인터페이스는 함수의 껍데기만 있는데, 이 함수들의 구현을 강제로 하기 위함!
- 추상 클래스는 상속 계층 구조에서 공통적인 코드를 공유하고, 인터페이스는 서로 다른 클래스 간에 일관된 동작을 보장하는 데 사용됩니다.

인터페이스 : 하나의 규약, 즉 구체적인 약속 같은 것, 모든 메서드가 추상 메서드

추상 클래스 : 미완성 메서드(추상 메소드)를 포함하고 있는 클래스 

- 인스턴스를 생성할 수 없고, 상속을 통해서, 자손 클래스에 의해서만 완성될 수 있다.
- 상속을 강제하기 위함, 부모 클래스에서는 정의만 해놓고, 실제 동작은 자식 클래스에서 하게 된다. 

</details>

### 클래스 패스와 클래스 로더란?

<details>
   <summary> 준비 답안 보기 (👈 Click)</summary>
<br />

클래스 패스는 클래스 파일을 찾는 데 사용되는 경로 목록이며, 클래스 로더는 클래스를 동적으로 로드하고 초기화하고 링크하는 데 사용된다.


- 클래스 패스 : 자바 컴파일러 및 런타임 환경이 클래스 파일을 찾는 위치를 지정하는데 사용한다.
- 클래스 로더 : 런타임(바이트 코드를 실행할 때) 환경에서 클래스 파일을 동적으로 로드하는 데 사용된다. 자바 클래스 파일을 JVM 내로 동적으로 로드하는 역할

  - 로딩 : 자바 바이트 코드(.class)를 메소드 영역에 저장한다.
  - 링크 : 읽어들인 클래스가 잘 구성되어 있는지 검사한다.
  - 초기화 : 클래스 변수들을 적절한 값으로 초기화한다. 즉, static 필드들이 설정된 값으로 초기화한다.

### 자바 Generic이란? Generic 사용 용도에 대해 설명하세요.

<details>
   <summary> 준비 답안 보기 (👈 Click)</summary>
<br />


- 타입을 일반화하여 코드의 재사용성과 안정성을 높이는 방법, 제네릭은 클래스나 메소드에서 사용할 내부 데이터 타입을 컴파일 시에 미리 지정면서 다음과 같은 장점을 가진다.
  1. 클래스나 메소드 내부에서 사용되는 객체의 타입 안정성을 높일 수 있다.
  2. 반환값에 대한 타입 변환 및 타입 검사에 들어가는 노력을 줄일 수 있다. 
- 클라이언트와 데이터 주고 받을 때, 컨트롤러 단에서 데이터 타입 모를 때 Generice를 이용해 T로 선언하면서 사용했다. 
- 컬렉션에서 제네릭을 사용하면 각 컬렉션에 저장되는 타입을 명확하게 지정할 수 있다. 이는 컬렉션 사용 시 타입 안전성을 보장하고 형변환을 줄여주는 등의 이점을 제공한다. 
- API 설계할 때 타입에 대한 제약을 명확하게 할 수 있다. 이는 API 사용자가 올바른 타입을 전달하도록 도와준다. 
- 이걸 왜 써?

  - T 타입으로 선언하고, 쭉 T 타입 연산만 하다가, T가 아닌 다른 값이 들어간다고 한다면,
  - 이것에 대해 프로그램이 실제 실행될 때 오류를 잡는게 아니라, 컴파일 단계에서 미리 오류를 알 수 있다.

### JVM이란? JVM의 동작 원리에 대해 설명하세요.

<details>
   <summary> 준비 답안 보기 (👈 Click)</summary>
<br />


- 자바 언어로 작성된 코드를 컴파일 하여 바이트 코드 형태로 변환하고, 이를 실행시킨다.

1. 컴파일 된 바이트 코드가 JVM 클래스 로더에 전달된다.
2. 클래스 로더는 동적 로딩을 통해 필요한 클래스를 로딩하고 링크해서, JVM 메모리(런타임 데이터 영역)에 올린다.
3. 실행엔진은 JVM 메모리에 올라온 바이트 코드를 명령어 단위로 하나씩 가져와서 실행한다.

### Array와 List의 차이에 대해 설명하세요.

<details>
   <summary> 준비 답안 보기 (👈 Click)</summary>
<br />


- Array와 List 모두 데이터 요소들을 저장하는 자료구조
- Array는 고정 크기를 가지고 있지만, List는 가변적인 크기를 가진다.
- Array의 크기는 생성 시에 결정되어 이후 변경할 수 없지만, List는 새로운 요소가 추가되거나 제거될 때마다 크기가 동적으로 조정된다.
- Array는 단일 데이터 타입만 저장할 수 있다. List는 서로 다른 데이터 타입을 저장할 수 있다. 다만, 제네릭을 이용하여 컴파일 시점에서 타입 안전성을 보장할 수 있다.
- Array는 연속된 메모리 공간에 저장되는 반면, List는 요소들이 연속적이지 않아도된다. 링크드 리스트로 구현 가능
- Array는 인덱스를 사용하여 빠르게 요소에 접근 가능, 반면에 List는 연결된 노드를 따라가야 하므로, 인덱스를 사용한 접근보다는 느리지만, 삽입과 삭제 연산에서 더 빠른 성능을 보인다. 

</details>

### 클래스와 객체의 차이에 대해 설명하세요.

<details>
   <summary> 준비 답안 보기 (👈 Click)</summary>

### 트랜젝션에 대해 설명하세요.

<details>
   <summary> 준비 답안 보기 (👈 Click)</summary>
<br />


- 하나의 작업을 수행하기 위해 필요한 데이터베이스의 연산을 모은 것으로, 데이터베이스에서 시행하는 작업의 논리적인 단위 입니다. (시작, 작업, 완료-commit, 롤백-)
- ACID (원자성, 일관성, 격리성, 지속성)

  - Atomicity (원자성) : 트랜잭션의 모든 작업이 전부 실행되거나, 전혀 실행되지 않아야 하는 것, 따라서 트랜잭션 내 작업 중 하나라도 실패하면, 전체 트랜잭션이 롤백되어야 한다.
  - Consistency (일관성) : 트랜잭션의 시작 전과 종료 후에도 데이터베이스는 일관된 상태여야 한다.
  - Isolation (격리성) : 다른 트랜잭션에 의해 실행 중인 작업이, 현재 트랜잭션에 영향을 미치지 않도록 격리되어야 한다.
  - Durability (지속성) : 성공적으로 완료된 트랜잭션의 결과는 시스템이 고장나더라도 영구적으로 데이터베이스에 반영된다.


트랜잭션은 다음과 같은 4가지 작업으로 구성됩니다.

1. 시작 (Begin): 트랜잭션을 시작합니다.
2. 작업 (Execute): 데이터베이스에서 원하는 작업을 수행합니다.
3. 완료 (Commit): 트랜잭션을 성공적으로 완료하고, 모든 변경 사항을 데이터베이스에 영구적으로 저장합니다.
4. 롤백 (Rollback): 트랜잭션을 취소하고, 트랜잭션 이전 상태로 되돌립니다.

따라서 트랜잭션은 데이터베이스에서 안전하게 작업을 수행하는 데 필수적입니다. 트랜잭션을 사용하면 데이터베이스의 일관성과 무결성을 유지하며, 데이터 손실을 방지할 수 있습니다.

</details>

### Inner join vs Outer join

<details>
   <summary> 준비 답안 보기 (👈 Click)</summary>
<br />


- 이너 조인 : 두 테이블 간의 공통된 값을 기반으로 레코드를 연결하는 것

  - 두 테이블에서 일치하는 값만 결과 집합에 반환한다.
  - 다른 조인 유형보다 더 효율적이며, 일반적으로 빠르다.

- 아우터 조인 : 하나 이상의 테이블에서 일치하는 레코드가 없는 경우에도 반환한다. 즉, 누락된 데이터를 보존하기 위해 사용된다.

  - left outer join : 왼쪽 테이블의 모든 레코드와 오른쪽 테이블의 일치하는 레코드 반환
  - right outer join : 오른쪽 테이블의 모든 레코드와 왼쪽 테이블의 일치하는 레코드 반환


<br>

이너 조인은 일치하는 레코드만 필요한 경우 사용하고, 아우터 조인은 일치하지 않는 레코드를 보전하거나 Null 값을 처리해야 하는 경우 사용된다. 

</details>

### 웹 서버와 WAS의 차이점에 대해 설명하세요.

<details>
   <summary> 준비 답안 보기 (👈 Click)</summary>

정적 페이지 : 바뀌지 않는 페이지

동적 페이지 : 인자에 따라 바뀌는 페이지


- 웹 서버 : HTTP 프로토콜을 사용하여 클라이언트로부터 요청을 받아 정적인 웹 페이지나 파일을 제공하는 소프트웨어 (Apache, Nginx)
- WAS (Web application Server) : 동적인 웹 페이지를 생성하고 실행하기 위한 미들웨어 소프트웨어 (Apache Tomcat, JBoss)

  - DB 조회 및 다양한 로직 처리 요구시 동적인 컨텐츠를 제공하기 위해 만들어진 애플리케이션 서버

- 웹 서버는 정적인 웹 페이지를 서비스하며, WAS는 동적인 웹 페이지를 생성하고 실행하는 데 필요한 기능을 제공함
- WAS = 웹 서버 + 웹 컴테이너

둘을 구분하는 이유?

- 웹 서버에서는 정적 컨텐츠만 처리하도록 기능 분배를 해서 서버 부담을 줄이는 것
- WAS를 통해 요청에 맞는 데이터를 DB에서 가져와 비즈니스 로직에 맞게 그때마다 결과를 만들고 제공하면서 자원을 효율적으로 사용할 수 있음
- 자원 이용의 효율성 및 장애 극복, 배포 및 유지 보수의 편의성 때문에 웹 서버와 WAS를 분리해서 사용하는 것이다.

</details>

### 세션과 쿠키의 차이점과 각각에 대해 설명하세요.

<details>
   <summary> 준비 답안 보기 (👈 Click)</summary>
<br />


- 쿠키와 세션 둘 다 서버와 클라이언트 간 정보를 유지하고 전달하기 위한 매커니즘
- 차이점
  1. 저장 위치 : 쿠키는 웹 브라우저에 저장, 세션은 서버에 저장
  2. 만료 시점 : 쿠키는 유효기간을 설정하여 지정된 기간동안 브라우저에서 유지된다. 세션은 클라이언트가 웹 사이트를 사용하는 동안에 유지된다. 브라우저를 닫거나 로그아웃하면 세션이 종료된다.
  3. 리소스 : 쿠키는 클라이언트에 저장되고 클라이언트 메모리를 사용, 세션은 서버에 저장되고 서버 메모리로 로딩되기 때문에 세션이 생길 때마다 서버 리소스를 차지함
  4. 보안 : 쿠키는 클라이언트 측에 저장되기 때문에 보안에 취약, 쿠키를 탈취하면 내부 정보를 볼 수 있음, 세션은 서버에 있기 때문에 상대적으로 안전
  5. 용량 제한 : 쿠키는 도메인별로 제한이 있음, 세션은 클라이언트가 접속하면 서버에 의해 생성되므로 제한이 없음

</details>

- 사용자의 HTTP 요청을 받아들이고, 서버 측 자바 코드를 실행하여 동적으로 웹 페이지를 생성한다. 이것은 자바 웹 어플리케이션을 위한 실행 환경을 제공하며, 동적 웹 사이트를 개발하기 위한 핵심 요소 중 하나이다.
- 아파치 : 웹 서벌 불리며, 정적인 데이터만 처리한다(HTML, CSS, 이미지 등)
- 톰캣 : 동적인 웹을 만들기 위한 웹 컨테이너, 서블릿 컨테이너라고 불리며, 웹 서버에서 정적으로 처리해야할 데이터를 제외한 JSP, ASP, PHP 등 을 웹 컨테이너(톰캣)에 전달한다.



아파치만 쓰면 정적인 웹 페이지만 처리하므로 처리 속도가 매우 빠르고 안정적이다. 하지만 톰캣(WAS)을 쓰면 동적인 데이터 처리가 가능하다. DB 연결, 데이터 조작, 다른 응용프로그램과 상호 작용이 가능하다.
